测试政策

我们知识库的这一部分概述了我们的综合测试策略，该策略遵循 左移 发展方式。与传统的不同 瀑布 模型，其中测试发生在编码之后，我们的策略强调在开发的最早阶段进行测试。这种积极主动的方法确保开发团队从项目一开始就考虑进行测试。

在整个开发和验证阶段，我们采用各种类型的测试来确保我们软件的稳健性和可靠性：

单元测试

单元测试是我们测试策略的基础层。他们细粒度地审查软件行为，单独检查各个组件。我们优先考虑单元测试，因为与其他类型的测试相比，它更快、更直接且更具成本效益。我们的开发团队在开发过程中编写单元测试。在大多数情况下，代码中的新功能及其关联的单元测试位于同一个 Pull Request 中，甚至在同一个提交中。为了将 Pull Request 合并到主开发分支中，所有单元测试（旧的和新的）都必须成功。

我们的单元测试方法的关键方面：

  • 关注需求和行为：

    我们的单元测试主要评估软件是否满足其指定的要求并正常运行，而不是内部设计验证。我们的大多数单元测试都被设计为“社交单元测试”，这意味着它们与实现细节无关，并且仅依赖于接口。这种方法允许我们在不影响测试的情况下重构代码，并最大限度地减少测试的不稳定性。然而，我们还为遗留组件维护一些传统的单元测试。

  • 测试快乐、糟糕和边缘路径：

    除了测试预期行为之外，我们的单元测试还包括探索各种场景（包括极端情况）的测试用例。除此之外，我们鼓励不仅测试“快乐路径”，还鼓励测试不利和容易出错的路径（坏路径和死亡路径）。

  • AAA 模式（安排-执行-断言）：

    只要有可能，我们的单元测试都遵循 AAA 模式，以提高可读性和可维护性。

  • 多个小型测试与大型测试用例：

    与具有多个断言的大型测试用例相比，我们更喜欢多个小型单元测试。这种方法增强了测试调试和理解。

  • 强制性测试补充：

    每当开发人员引入新功能或错误修复时，他们都需要创建相应的单元测试。我们通过代码审查和代码覆盖率来强制执行这一点。

  • 组件标签：

    我们按组件标记单元测试，以便轻松评估每个软件组件的测试覆盖率。

我们主要使用 谷歌测试 对于单元和集成测试，尽管我们有一些测试是用纯 C++ 编写的，没有测试框架的帮助

集成和组件测试

集成测试包含一些子组件或完整的 ORAN 项目，以黑盒测试的方式验证输入输出关系。这些测试在（子）组件的初始开发阶段发挥着至关重要的作用，充当验证一般场景的骨架测试。

端到端 (E2E) 测试

考虑到我们生态系统的复杂性，我们仍然需要 E2E 和系统测试来评估我们的 CU/DU 解决方案与真实和模拟 UE 以及 5G 内核的行为和集成。这些测试主要由测试团队管理，并遵循两种不同的方法：

  • 无硬件测试：

    使用 ZMQ 和模拟器，我们开发了一个测试套件，其中包含附加、重新附加和各种类型的流量等基本场景。这些测试易于执行，不需要硬件设备，适合开发人员在重大更改后评估软件的行为。它们每晚运行，并且可以在 Pull 请求中触发。

  • 硬件测试：

    对于现实场景，我们采用商业 (COTS) UE 设备和/或 RU 来触发复杂的测试。这些测试评估行为、性能以及与外部组件的集成。它们每晚和每周运行一次，并提供有价值的指标和关键绩效指标 (KPI)。

为了促进这些测试，我们开发了一个内部 E2E 测试框架，该框架经过容器化以实现部署灵活性，确保测试复制生产环境。此外，我们还利用来自 维亚维 验证我们的产品。

虽然我们的 E2E 测试套件不会像单元测试那样随着每次代码更改而增长，但当引入重要功能或探索性测试识别出值得注意的自动化场景时，我们会不断添加新测试。

探索性测试

由于我们旨在支持的现实场景的复杂性，探索性测试在我们的策略中尤为重要。每当发现问题或意外行为时，我们都会尽力在尽可能低的级别创建测试，最好是单元测试。

源代码的机械性规范

源代码格式化

srsRAN Project 项目使用一组定义的格式化规则，并严重依赖 clang-format 来自动源代码格式化。 您必须在存储库的根目录中使用自定义 clang-format 文件，请参阅 这里 有关此文件中的选项的更多信息。

如果您对有关空格、最大宽度等的具体规则感兴趣，请检查上述文件的内容。

将新提交添加到存储库时，请确保您已使用 clang-format 格式化新更改 之前 提交它们。

注释

注释必须被视为代码的重要组成部分，因为它们能极大地方便审阅者和维护者。从这个角度来看，注释应说明代码要做什么以及为什么这样做，而不必陷入过于细碎的实现细节。最重要的是，要假定读者对当前主题未必像你一样熟悉。

下面只对开发者应如何编写注释做一个非常简要的说明。更完整的指南请参见 自生成文档。

注释指南

  • 注释应写成符合英文书面表达的句子，使用正确的大小写和标点，并以句号结尾。单词拼写应遵循美式英语规范，可参考 Merriam-Webster Dictionary。

  • 普通注释行应以双斜杠开头，文档注释行则应使用三斜杠。即使是多行注释，也始终应采用这种双斜杠/三斜杠格式。

/// \brief Does something.
///
/// More details about the function.
void some_func(){
  // Now we print a simple text.
  fmt::print("Some comment.");
}

  • 注释应始终另起一行，放在其所说明的代码之前。特别是，文档注释应写在类或函数声明之前，而不是写进函数体内部。普通注释也不应写成行尾注释，或脱离函数体单独放置。

// Some general comment outside the body of a function - WRONG!

/// Does something. - OK
void some_func(){
  // Now we print a simple text. - OK
  fmt::print("Some comment.");

  fmt::print("Something else."); // Side comment. - WRONG!
}

  • 如果注释内容本身需要体现层级缩进，建议根据缩进层级在开头使用一个或多个“大于号” (>)。例如，下面示例中的注释表明 PUCCH 配置从属于 UL 配置。

uplink_config srsran::config_helpers::make_default_ue_uplink_config(const cell_config_builder_params& params)
{
  // > UL configuration.
  uplink_config ul_config{};
  ul_config.init_ul_bwp.pucch_cfg.emplace();

  // >> PUCCH configuration.
  auto& pucch_cfg = ul_config.init_ul_bwp.pucch_cfg.value();

  // ... more code ...
}

  • 通常不允许使用 C 风格注释 /* */。唯一的例外是文件头部（见下文）以及函数调用中的参数说明注释，这在传递 bool 或 nullptr 之类的参数时尤其有用。

// Hard to see what "true" and "nullptr" actually refer to.
obj.method(a, b, true, nullptr);
// Easy to see what the input values actually do.
obj.method(a, b, /*enable_X=*/true, /*options=*/nullptr);

  • 合并请求中 不应 包含被注释掉的代码行。如果确实需要为了说明或临时保留调试打印，请使用 #if 0 和 #endif。这种方式可以正确嵌套，整体上也比 C 风格注释更稳妥。

头文件保护宏

所有头文件都应具备防止重复包含的保护机制。srsRAN Project 代码库使用 #pragma once 指令，这一写法已被主流编译器广泛支持。与传统的 include guard（通过 #ifndef 和 #define 实现）不同，它不需要唯一标识符，也不需要结尾的 #endif。

以下示例说明了这一点：

/*
 * File header...
 */

#pragma once

#include "foo.h"
#include <file.h>

namespace srsgnb {
// ...
} // srsgnb

#include 样式

尝试包含最少的列表 #include 并在依赖项发生变化时清除冗余头文件。为此， 可以利用头文件的传递包含特性。例如，如果 foo.h 已经包含了 bar.h，那么在需要使用这两个文件中定义的函数或类时，仅包含 foo.h 即可。

包含列表应紧接在头文件注释之后，如果处理头文件，则应在包含防护之后。 包含文件应按以下顺序列出：

  1. 主模块头。

  2. 本地和私有标头。

  3. srsRAN Project 或子项目标头（srsgnb/…、srsue/…、srsran/…等）。

  4. 系统库包括。

保持每个类别按完整路径按字典顺序排序，并避免在类别之间添加换行符或包含指令。 主模块头应该始终位于列表中的第一个。子项目标头应包含在 srsran 标头之前（从最具体到最不具体，例如：srsran 之前的 srsgnb）。

#include "my_class_header.h"            // category 1
#include "private_module_utils.h"       // category 2
#include "srsenb/hdr/public_header.h"   // category 3
#include "srsran/adt/bounded_vector.h"  // category 3
#include <string>                       // category 4
#include <vector>                       // category 4

在C++文件中使用C++库头文件而不是使用C库头文件，例如： cstring, cassert, cstdint等

最后一点， clang-format 将自动为您按字典顺序对所有包含文件进行排序。

语言与编译器

将编译器警告视为错误

避免提交生成编译器警告的代码。有时您可能会收到误报警告，在这种情况下找到一种方法来抑制它。

代码可移植性

尝试在所有情况下编写可移植的代码。如果您遇到需要做一些不可移植的事情的情况， 然后将其放在记录的界面后面，以便它集中在一个地方，而不是分散在不同的地方。

避免 RTTI 和异常

在 C++ 代码中， 不 使用异常或 RTTI（运行时类型信息，例如：dynamic_cast<>).

使用 auto

一般来说，当 auto 使代码更具可读性且更易于维护时，请使用 auto。典型的情况是迭代器或复杂的模板类型。 一些例子：

auto *p = std::unique_ptr<int>(new int);  // good: easy to infer that p is an int*
auto i = my_map.begin();                  // good, easy to see it is an iterator and avoids writing std::unordered_map<std::string, int>::iterator
auto a = my_func();                       // bad: what type does my_func return?

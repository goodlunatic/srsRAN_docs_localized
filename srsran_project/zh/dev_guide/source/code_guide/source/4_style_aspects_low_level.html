风格方面：低级问题

命名约定

名称应遵循以下编码 蛇箱 模式。 对于变量、函数和类，名称模式始终为小写，并用下划线分隔单词。 SI 单位和 其他带有大写字母的常用单位（dB、dBm）应保留其原始形式。 名称应具有描述性且易于阅读，并避免使用缩写，除非众所周知。

int n_pdu          // Not so good, it's difficult to imagine what represents.
int number_of_pdu  // Good, descriptive name.
int pdu_count      // Good, descriptive name.
int path_loss_dB   // Good, descriptive name with unit suffix.

除迭代器外，应避免使用非描述性名称 for 非常小的范围内的循环或变量。

for (unsigned i = 0, e = size; i != e; ++i) { // i and e are iterators inside a for loop.
  std::lock_guard<std::mutex> lck(mutex);     // Good as the scope is only 2 lines long.
  detach(i);
}

名称应遵循以下规则（始终使用 Snake_case）：

  • 类型名称 （类、结构、枚举、typedef……）应该是名词。

  • 变量名 应该是名词（它们代表状态）。

  • 函数名称 应该是动词短语（它们代表动作），并且类似命令的功能应该是命令式的。

  • 枚举声明 是类型，因此它们应该遵循类型的命名约定。

  • 枚举器 （例如： 枚举 { 红，绿 }) 应遵循类型的命名约定。作为方便常量的枚举数应以大写形式编写。例如：

enum {
  MAX_ELEMENTS = 32,
  BYTES_PER_ELEMENT = 64
};

断言和期望

srsRAN Project 提供了一个名为的自定义断言宏 srsran_assert。 尽可能多地使用它来检查所有先决条件和假设。这将有助于减少调试时间 断言可能由您的代码甚至外部错误代码触发。

确保在断言语句中写入描述性错误消息，该消息将在触发断言时打印。 例如：

char *get(unsigned idx) {
  srsran_assert(idx < v.size() && "get() out of range!");
  return v[idx];
}

其他附加示例：

srsran_assert(buffer->is_valid() && "Buffer should always be valid!");
srsran_assert((channel == DLSCH || channel == ULSCH) && "Channel type is invalid!");
srsran_assert(idx < get_num_ues() && "UE index value is out of range!");
srsran_assert(v1.size() == v2.size() && "vector sizes must be identical!");

注意事项

如果用户输入可以触发错误条件，则不要使用断言，而是使用可恢复的错误机制。

警告

哪个？

使用断言的另一个好处是，您可以对许多接口应用“契约设计”方法，从而帮助减少很多 错误检查和错误处理混乱。由于软件通常是分层的，因此您可以验证特定层中的用户输入，并将其集中在一个位置 然后通过断言的方式让内层接口通过契约来设计。使用这种方法将有利于更简单的界面的出现， 减少极端情况，并且实现将需要验证更少的状态，这将减少未经测试的极端情况中的错误。

使用断言时，您可能会收到以下警告 “未使用的价值” 如果断言被禁用（主要在发布版本中）。例如：

unsigned size = v.size();
srsran_assert(size > 42 && "Vector smaller than it should be");

bool is_value_new = set.insert(x);
srsran_assert(is_value_new && "The value shouldn't be in the set yet");

在第一种情况下，调用 v.size() 仅在断言中有用，如果断言被禁用，我们不希望它执行。在这种情况下，代码应该移动到断言内部。 在第二种情况下，无论断言是否启用，调用的副作用都一定会发生。在这种情况下，该值应转换为 void 以禁用警告。

srsran_assert(v.size() > 42 && "Vector smaller than it should be");

bool is_value_new = set.insert(x);
(void)is_new_value;
srsran_assert(is_value_new && "The value shouldn't be in the set yet");

警告

写一些关于期望的东西

请勿使用 using namespace std

当您需要引用标准库中的标识符时，更愿意显式使用 std:: 前缀而不是依赖 using namespace std;。 在头文件中，添加 using 命名空间指令将污染包含此头文件的任何源文件的命名空间，从而导致维护问题。

此规则的例外情况（不适用于 std 命名空间）用于实现文件（.cpp）。例如，srsRAN Project 中的所有代码都实现了位于 srsran 命名空间。在这种情况下，更清楚的是，.cpp 文件在文件顶部、包含列表之后有一个 using namespace srsran 指令。 这将减少文件正文中的缩进。

此规则的一般形式是，在任何命名空间中实现代码的任何 .cpp 文件都可以使用该命名空间（包括其父命名空间），但不应使用任何其他命名空间。

使用范围循环

自从 C++11 中引入基于范围的 for 循环以来，很少需要对迭代器进行任何类型的显式操作。尝试使用基于范围的 for 循环 尽可能，例如：

for (const auto &ue : ue_db)
  ... use ue ...

循环结构

如果无法使用基于范围的 for 循环，并且需要编写显式的基于迭代器的循环，请注意是否在每次循环迭代时重新评估 end()。 最常见的错误是以这种方式编写循环：

for (auto i = x.begin(); i != x.end(); ++i)
  ... use i ...

问题是它评估 x.end() 在每次迭代中。相反，请使用在循环开始之前对其求值一次的循环。这可以使用以下表格来完成：

for (auto i = x.begin(), e = x.end(); i != e; ++i)
  ... use i ...

注意事项

这两个循环具有不同的语义：如果容器在循环内发生变化， x.end() 每次循环时都可能改变它的值， 所以第二种形式可能不正确。如果您确实依赖于这种行为，请在第一种形式中编写循环并添加一条注释来表明您这样做了 是故意的。

通过一致使用第二种形式，读者将隐式地看到循环不会改变容器，而无需分析循环体， 使代码更容易阅读和理解。

一般来说，循环中迭代器比较的 C++ 语法是使用 != 运算符而不是 <。这应该一致地使用，即使 迭代器是一个普通整数。

例如，首选形式如下所示：

for (unsigned i = 0, e = x.size(); i != e; ++i)
  ... use i ...

相反 C风格的方式：

for (unsigned i = 0, e = x.size(); i < e; ++i)
  ... use i ...

使用预自增

在 C++ 中（不适用于 C 代码），预自增 (++x) 可能不会慢于后增量 (x++）并且可能比它快得多。 因此，尽可能优先使用预增量。

匿名命名空间的使用

使用匿名命名空间使其周围的内容对文件私有，就像 static用于 C 函数和全局变量。

匿名名称空间的问题在于它们鼓励其主体缩进，并且减少了引用的局部性：如果您看到 C++ 文件中的随机函数定义，很容易看出它是否被标记为静态，但查看它是否在匿名名称空间内可能会 需要扫描文件的一大块。

由于这些原因，请遵循这个简单的规则：使匿名名称空间尽可能小，并且仅将它们用于类声明， 不 对于功能。

namespace {
class foo { // foo class has internal linkage
  ...
public:
  int bar(); // define this method outside the anonymous namespace.
  ...
};
} // end anonymous namespace

static void some_helper() { // static function marked as static outside anonymous namespace.
  ...
}

int foo::bar() { // method definition outside of anonymous namespace to keep namespace as compact as possible.
  return 42;
}

使用 C++ 类型转换

避免 在 C++ 代码中使用 C 风格的转换。

C++ 强制转换在代码中比 C 对应的强制转换更易于阅读。另一个原因是C++有四种不同的 类型转换，允许程序员根据情况选择正确的工具。虽然 C 演员可能 适用于任何情况，并可能禁用编译器发出的任何警告。

自生成文档

srsRAN Project 存储库使用 多氧 直接从带注释的 C++ 源文件生成 API 文档。为了为API做出贡献 由于文档风格一致，以下部分提供了一些针对最常见代码元素的一般准则。请参考 的 Doxygen 文档 有关 Doxygen 命令和功能的完整概述。

一般方面

该文档必须以英语书写，并采用美国英语拼写，如以下主要条目所示： 韦氏词典。 对于所有编辑事项（例如，首字母缩略词、复数、大写、方程式），建议的参考文献是 IEEE 作者编辑风格手册 ，特别是 E节 其中。

所有代码实体都在实体声明之前使用注释块进行记录。将文档块放在同一行 不允许使用 code 元素，即使是简短的一行注释。如前所述 这里, 文档注释的所有行 块以三斜线开头。另外，srsRAN Project 文档更喜欢 \命令 Doxygen 命令的形式（与 @命令).

通常，文档块由以命令开头的简短描述（理想情况下不超过一行）组成 \简介 和一个或 更多段落由空行分隔。当没有提供详细描述时， \简介 简单描述中的命令可以是 省略。简短的描述应该开门见山，不要使用“The 我的班级 类提供/指定......”（更多 关于这一点如下）。

/// \brief 代码元素的简要描述。
///
/// 详细描述从这里开始。这可以随机跨越多行
/// 填写此示例。 Lorem ipsum dolor sat amet consectetur adipiscing
/// elit lacinia maecenas, hendrerit luctus libero vivamus elementum feugiat
/// 扭矩累积到 eleifend，diam orci aptenttincidunt a iaculis sed nisi。
///
/// 更详细的描述。段落之间用空行分隔。库拉埃
/// arcu tempor urna convallis pulvinar conubia rutrum auctor，rhoncus nam
/// faucibus montes velit 非骚扰，nostra proin metus senectus sem tempus
/// 令人兴奋。
类 例子1;

/// 代码元素的简短且唯一的描述。
类 例子2;

所有代码实体在声明时都应记录下来。定义时不需要（实际上，不鼓励）重复文档块 代码实体。特别是，虚拟方法仅在声明接口/基类时才记录，并且在声明接口/基类时不会重复记录 定义实施。

其他一般性建议如下

  • 当引用技术规范或类似文件时，仅 参考特定的部分和表格，因为它们之间不会改变 发布，并避免使用页码。考虑包括以下内容的摘录 文件（如果他们提供澄清）。

/// 该类实现 TS38.211 第 4.2.1 节。

  • 在代码中查找有关首字母缩略词拼写的示例以及 参考文献。例如，推荐的形式是 TS38.211（注意缺少 TS 和号码之间的空格）或 DM-RS。

  • 只要有可能，对公式使用严格的数学符号， 方程和相关问题。例如， \((0, 1)\) 表示开区间 0 到 1 之间（即所有实数 \(x\) 这样 \(0 < x < 1\)），同时 \([0, 1]\) 代表闭区间，包括两个端点。半开 间隔，例如 \([0, 1)\) 或 \((0, 1]\)，也是可以的。离散集是 用枚举符号写成（例如， \(\{1, 4, 23\}\))，可能有省略号 当含义很明显时（例如， \(\{1, ..., 10\}\) 前十名 自然数，或 \(\{1, 1.2, 1.4, ..., 2.4\}\) 对于 1 到 1 之间的所有数字 2.4，增量为 0.2）。

  • 将代码示例包裹在之间 \代码 和 \结束码 命令。

/// \代码
类 示例_类 {
  整数 无用字段;
  };
/// \结束码

  • 使用 \c 命令或 <tt>...</tt> 标签为 短的内联代码位，例如 \c 变量名 或 <TT> 变量名 = 5 </tt.

  • 开始一个段落 \注意, \警告 或 \备注 如果你想额外强调它（具有明显的含义）。

/// \warning 如果输入序列为空，将会抛出异常。

文件

可以在版权标头之后提供文件内容的描述。特别推荐用于测试和的源文件 应用程序。对于文件来说， \简介 始终需要命令。

/*
 *
 * 版权标题...
 *
 */

/// \文件
/// \brief LDPC 编码器和解码器的单元测试。
///
/// 对于所有可能的基本图和提升尺寸，测试从
/// 归档一小组消息和相应的代码块。这些消息是
/// 馈送到编码器，编码器的输出与代码块进行比较。同样，
/// 代码块被馈送到解码器，结果消息是
/// 与示例相比。

类和结构

类和结构文档应该提供关于它代表什么以及如何使用类的实例化的足够信息。在简短的 描述时，不要写“类/结构代表/定义”之类的表达式，而是直接提供更有意义的信息。使用 \简介 命令，当简短描述后跟详细描述时（如果文档仅限于简短描述，则可选）。

明确描述类的边缘情况、副作用和不太明显的方面是一个很好的做法（请参阅最后一个示例）。

/// LDPC 速率解匹配器接口。
类 ldpc_rate_dematcher;

/// 解码器配置。
结构体 配置;

/// \brief PHY&ndash;FAPI 双向适配器接口。
///
/// 该适配器是将 FAPI 消息转换为接口的集合
/// 它们的 PHY 层对应部分，反之亦然。
///

类方法和自由函数

方法和函数对应于动作。因此，简短描述通常以动词（第三单人称）开头。

/// \brief 查找大于\c n 的最小素数。
未签名 素数大于(未签名 n);

上面例子中的 free 函数非常简单，输入和输出一目了然。对于更复杂的情况，更愿意提供更多 信息通过 \参数 和 \返回 命令。参数描述应遵循与一般变量相同的准则（参见 下一节）。

此外，与类和结构的解释类似，应该指出边缘情况和不可预测的行为。

/// \brief 解码代码块。

/// 返回一个空值（即，CRC 失败）并将所有输出位设置为 1。
虚拟的 可选的<未签名> 解码(位缓冲区& 输出,
                                  跨度<常量 对数似然比> 输入,
                                  crc_计算器* CRC,
                                  常量 配置& cfg) = 0;

类数据成员、对象、变量

对象和类数据成员，无论是变量还是常量，静态与否，都被视为它们所代表的概念。因此，他们的简短 描述不应显示“代表”、“指示”、“表示”或类似术语。另外，不需要重复对象的类型， 因为 Doxygen 重复对象的声明及其描述。尽管不常见，但此类文档 实体可能还需要一段或多段详细描述。下面报告了一些很好的文档示例。

/// 最大迭代次数。

布尔值 新数据 = 真实;

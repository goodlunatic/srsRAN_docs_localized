建议

函数和类长度

避免 编写太大的类或函数。

根据经验，函数应该适合在一个窗口中，并且不需要滚动来查看它们。 尝试编写只执行一件事或操作的函数。如果您编写的函数执行不止一件事，请尝试将其拆分为多个函数， 每个都做一件事情。这将极大地提高代码的可维护性。通过使用简洁的函数名称， 通过详细记录每个函数，代码将更容易理解。

对于类来说，设置最大行数比较困难，因此，给一个类一个单一的职责。虽然责任观念 听起来可能很抽象，但另一种思考方式是，一个类应该只有“一个改变的理由”。 如果一个类做多项事情或职责，最好将其拆分为多个类。 这样我们就可以避免陷入 神级 反模式可能是维护的噩梦。

适用范围

定义要使用的变量，使其作用域尽可能短。这适用于 C++ 和 C 代码。

例如：

整数 我;
浮动 f;
...
... 其他 无关的 代码 ...
...
f = 获取浮动();
一个 = f * 获取乘数();

应改写为：

...
... 其他 无关的 代码 ...
...
浮动 f = 获取浮动();
整数 我 = f * 获取乘数();

逻辑运算符

代码库使用逻辑运算符的标准形式：

-&&, || 和 !

与替代形式相反：

  • 和, 或 和 不

以下代码块显示了逻辑运算符的正确和错误使用：

如果 (!一个 && (乙 == 8))     // 正确，仅使用标准形式。

如果 (不 一个 && (乙 == 8))  // 不正确，使用了“not”而不是标准形式“!”。

在指针上使用引用(&)(*)

一般来说，最好使用引用而不是指针。 C++ 引用的语法比指针更简单，并且始终引用有效的对象（除非您执行一些未定义的行为）。

当指向的对象可以选择时，指针非常有用 空指针，但是管理这种情况可能需要添加额外的检查 来管理这种情况。

静态的 布尔值 检查对象是否相等(富 *左, 富 *对);              // 不好，左或右可能是 nullptr，更喜欢引用。
静态的 布尔值 检查对象是否相等(富 左, 富 对);                // 不好，复制对象。
静态的 布尔值 检查对象是否相等(富 &左, 富 &对);              // 不好，缺少 const，检查对象不应修改它们。
静态的 布尔值 检查对象是否相等(常量 富 &左, 常量 富 &对);  // 好。
静态的 无效 填充可选对象(富 *x);                                // 很好，x 是可选的，函数检查 nullptr。

常量正确性

编写代码是 常量 正确。

常量 对象& 获取对象() 常量; // 返回值无法修改，方法不会修改类的内容。

避免复杂的表达式

尝试使用局部变量来存储中间结果或在多次调用函数时缓存函数调用的结果 众所周知，这一点不会改变。这将有助于避免复杂的表达式。

例如：

为了 (未签名 我 = 0; 我 != 获取条数(); ++我) {
  如果 (获取条数() < 我) {
    做某事();
  }
  做其他();
}

可以转换为这段代码，从而节省了许多调用 get_bar_count()

未签名 条数 = 获取条数();
为了 (未签名 我 = 0; 我 != 条数; ++我) {
  如果 (条数 < 我) {
    做某事();
  }
  做其他();
}

在某些情况下，您可能 需要 处理复杂且深层的嵌套结构。在这些情况下尝试存储中间成员 作为指针或引用（如果编写 C++ 代码）并记住使用 常量 如果只读取值。 这将避免不必要的输入，拥有更紧凑的代码，并可能提高不必要的间接追逐指针的性能。

例如：

为了 (uint32_t j = 0, 杰 = 指标_tmp.尺寸(); j != 杰; ++j) {
  指标[j].dl.n_样本 += 指标_tmp[j].dl.n_样本;
  指标[j].dl.MCS += 指标_tmp[j].dl.n_样本 * 指标_tmp[j].dl.MCS;
  指标[j].乌尔.n_样本 += 指标_tmp[j].乌尔.n_样本;
}

可以转化为：

为了 (uint32_t j = 0, 杰 = 指标_tmp.尺寸(); j != 杰; ++j) {
  下行链路指标 &dl = 指标[j].dl;  // 存储非常量引用来访问结构体
  常量 下行链路指标 &dl_tmp = 指标_tmp[j].dl;  // const ref 来访问结构体

  dl.n_样本 += dl_tmp.n_样本;
  dl.MCS += dl_tmp.n_样本 * dl_tmp.MCS;

  常量 上行链路指标 &ul_tmp = 指标_tmp[j].dl;
  乌尔.n_样本 += ul_tmp.n_样本;
}

神奇数字

避免使用幻数。相反，使用变量 (静态的, 常量表达式等）：这样，你将被迫给出它 描述性名称，提高代码可读性。

如果 (乙 < 2.0)  // 不好
  ...

静态的 常量表达式 浮动 检测阈值dB = 2.0;
如果 (乙 < 检测阈值dB)  // 很好 - 我们现在知道这是一个以 dB 为单位的检测阈值
  ...

固定宽度整数类型

使用固定宽度整数类型 (uint32_t等...）中发现 cstdint 和 stdint.h 需要时使用头文件 确保变量需要具有固定大小，独立于目标架构。

通常，在定义消息结构、硬件接口、网络编程等时，您需要使用这些类型…… 然而， 避免 当类型宽度不重要时使用它们，因为固定大小类型的使用是由程序员完成的声明 受影响的代码需要特殊处理和仔细处理。

请注意，所有 32 位及以上架构都定义了 整数 和 未签名 整数 是32位的，所以使用通用的是安全的 安全地键入，除非您知道需要更宽的类型。

避免使用 长 类型，因为它可以是 32 位或 64 位，具体取决于体系结构。请参阅 标准定义 了解更多信息。

函数签名

声明函数时，参数列表应按如下顺序排列：

  1. 通过引用或指针传递的参数（如果有）仅用于存储函数输出。这些参数由 Doxygen 命令记录 \参数[输出].

  2. 通过引用或指针传递的参数（如果有）包含函数的输入，并且也将由函数本身修改。这些参数由 Doxygen 命令记录 \参数[输入，输出].

  3. 按值、常量指针或常量引用传递的参数（如果有），表示函数的输入。这些参数由 Doxygen 命令记录 \参数[输入]。特别是，收集许多配置参数的结构应出现在参数列表的最后。

例如：

/// \brief 做某事。
///
/// 该函数写入一个跨度，读取并修改一个向量，仅读取其余参数。
/// \param[out] fnct_out 主函数输出（回想一下 \c span 的行为类似于指针）。
/// \param[in,out] my_vector 函数可以读取和修改的整数向量。
/// \param[in] another_vector 函数只能读取的整数向量。
/// \param[in]数量 函数需要了解的另一个值。
/// \param[in] cfg_struct 描述函数工作上下文的结构。
无效 我的函数(跨度<整数> fnct_输出, 标准::矢量<整数>& 我的向量, 常量 标准::矢量<整数>& 另一个向量, 整数 数量, 常量 配置_t& cfg_结构体);

类布局示例

本节描述了类声明的推荐布局，以及内联的建议。

/// 类描述块。
类 我的班级 : 公开的 接口_1 {
    // 首先定义私有常量可配置参数 - 类维护者很容易找到它们始终位于
    //同一个地方。
    /// 常量1解释。
    静态的 常量表达式 未签名 一些_常量_1 = 2;
    /// 常量2解释。
    静态的 常量表达式 未签名 一些_常量_2 = 7;

公开的:
    // 首先声明/定义特殊成员（构造函数、析构函数等...）
    // 不需要记录特殊成员，除非有些不重要的事情需要解释。
    我的班级() { ... }  // 如果ctor主体很大，则将其下移到cpp文件中。
    ~我的班级() { ... }  // 同样，如果dtor主体很大，则将其下移到cpp文件中。

    /// 方法 1 解释块。
    无效 方法1();
    // 使用单个换行符分隔方法声明。
    /// 方法 2 解释块。
    无效 方法2();

    // 尝试将同一类别的方法、访问器、修饰符、实用程序分组...

    /// Getter 解释块。
    某事& 得到一些东西();
    常量 某事& 得到一些东西() 常量; // 随意提供两个相同的解释块或声明
                                            // 连续两个方法，这里不需要换行。

    /// Getter 解释块。
    另一种事物& 获取另一个东西();  // 这里我们分离了 const 和非 const 方法声明。
                                        // 请在每个班级基础上保持一致。

    /// Getter 解释块。
    常量 另一种事物& 获取另一个东西() 常量;

    // 无需为重写的接口方法添加文档，因为它们已经记录在接口中，
    // 我们不想保留文档块的两个副本，因为它们很容易产生分歧，从而产生维护问题。
    无效 接口方法1() 覆盖;
    // 请记住在方法之间添加换行符。
    无效 接口方法2() 覆盖;

    /// 枚举描述块。
    // 在将要使用的站点附近定义枚举，以在读取界面时提高局部性。
    枚举 类 我的枚举 {
        枚举器1,
        枚举器2
    };

    /// 使用上述枚举描述块的方法。
    无效 方法使用枚举(我的枚举 e); // my_enum 已在上面声明 - 很容易找到。

受保护的:  // 公共方法部分完成后，声明其余部分（如果适用）（受保护的、私有的）。
    // 这里的受众已经变成了类开发者，不再是公共接口用户，我们避免污染公众
    // 带有实现细节的接口
    /// 受保护方法解释块。
    无效 保护方法();

私人的:  // 私有方法部分。
    // 只有需要了解实现细节的用户才会阅读此内容。远离公共接口。
    /// 私有方法解释块。
    无效 私有方法();

受保护的: // 最后声明成员变量。将它们与方法分开。
    /// 受保护的成员变量。
    未签名 受保护的成员变量;

私人的: // 最后，声明私有成员变量。
    // 变量名应该是描述性的，所以大多数时候不需要记录它们。
    未签名 成员_var_1;
    无效* 成员_var_2;
    /// 保护对 memb_var_2 的访问。
    // 对于互斥体，它有助于添加一个关于保护内容的简短解释...
    可变的 标准::互斥体 米;
};

故障排除

COTS UE 问题

以下是将 COTS UE 与 srsENB 结合使用时最常见的问题。可以找到关于此的完整应用说明 这里。有关以下问题的更多详细信息，请参阅此内容。

UE 看不到网络

UE 看不到网络的最可能原因是 eNB/EPC 配置、RF 条件以及所使用的 RF 前端的频率精度。

首先要检查的是您所配置的 LTE 频段和 EARFCN 是否受您所使用的 UE 支持。大多数 UE 设备支持分配给 LTE 的频段子集。还要确保配置的 LTE 载波的完整带宽位于您正在使用的频段内。

某些 UE 设备无法查看使用测试 PLMN MCC/MNC 值配置的网络。例如， 常用的 MCC/MNC 值 999/70、901/70 或 001/01 可能不起作用，特别是对于使用 Intel 基带芯片组的 iPhone 设备。相反，请尝试将网络的 MCC 设置为您所在国家/地区的特定值 （例如爱尔兰为 272）。可以找到 MCC 代码的完整列表 这里。跨国公司 然后可以将值设置为您所在国家/地区的移动网络运营商当前未使用的任何值。

RF 条件可能会受到所使用的天线的影响，我们建议 垂直2450 Ettus 的天线（或类似产品）。确保天线彼此成 90° 角放置，以最大限度地减少串扰。 如果可能，您应该使用频谱分析仪或其他此类设备来检查 RF 硬件传输的信号的质量。如果信号太弱或格式错误，则 UE 可能无法成功接收信号并且不会尝试连接。的 gr-fosphor工具 是一款非常有用的SDR频谱分析仪，可用于检查传输的RF信号的属性。

RF 前端的载波频率精度较低也可能导致 UE 无法发现网络。如果不使用外部参考时钟输入，大多数 SDR 前端的时钟精度都相当低。可以使用实验室设备或开源工具，例如 kalibrate-rtl 估计 RF 前端的 CFO，并通过在 eNodeB 配置文件的下行链路和上行链路载波频率设置中设置较小的频率偏移来手动补偿。

UE 无法连接

如果 UE 发现网络但无法成功连接，您可以使用 Wireshark 检查 srsENB 提供的 MAC 层 PCAP，以查看连接过程中的哪一点失败。有关 MAC 层 PCAP 和使用 Wireshark 的更多信息，请参阅 这里 在文档中。

无法访问互联网

如果连接的 UE 无法访问互联网，这可能是由于 UE 和/或 eNB 中的 APN 配置错误。请参阅 应用笔记 有关如何配置此信息的信息。

另一个常见原因是 EPC 的 IP 路由配置错误。如果使用 srsEPC，请确保遵循 IP 伪装中的说明 应用笔记.

峰值吞吐量

由于多种不同原因，可实现的最大 srsENB 峰值吞吐量可能会受到限制。其中包括所使用的 PC、网络配置、RF 硬件和物理网络条件的限制。

计算能力

为了实现峰值吞吐量，我们建议使用配备第 8 代 i7 处理器或更高版本、运行 Ubuntu 16.04 操作系统或更高版本的 PC。规格较低的机器也可以成功运行 srsENB，但最大可实现吞吐量较低。

PC 的 CPU 调速器应设置为性能模式，以实现最大的计算能力和吞吐量。例如，在 Ubuntu 上可以通过以下方式进行配置：

echo "performance" | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor

同样，您还应该确保您的 SDR 驱动程序是最新的，并且您正在通过 USB 3.0 运行，因为这也会影响最大吞吐量。

如果使用笔记本电脑，用户在运行 srsENB 时应始终保持 PC 连接到电源，因为这将避免由于计算机上的 CPU 频率调整而导致性能损失。

srsENB 应用程序的计算要求与所使用的 LTE 载波的带宽密切相关。例如，使用 100-PRB 载波的最大吞吐量将需要比使用 25-PRB 载波的最大吞吐量更强大的 CPU。如果您的计算机不够强大，无法支持具有给定网络配置的 srsENB，您将看到来自 SDR 前端的延迟和/或溢出数据包报告。

RF 硬件

RF 信号本身也会影响网络可以实现的峰值吞吐量。确保所使用的无线电已正确校准并使用适当的增益设置。可以使用 srsENB 的控制台跟踪输出来快速检查 RF 信号的运行状况。
